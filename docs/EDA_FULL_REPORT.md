## 1. 데이터 개요

| 파일 | 행 | 열 | 설명 |
|------|-----|-----|------|
| `train.csv` | 250,000 | 94 | 10,000 시나리오 × 25 타임슬롯 |
| `test.csv` | 50,000 | 93 | 타겟 컬럼 없음 |
| `layout_info.csv` | 300 | 15 | 창고 레이아웃 보조 정보 |

- **피처 구성**: 90개 수치형 + ID 3개 (`ID`, `layout_id`, `scenario_id`)
- **타겟**: `avg_delay_minutes_next_30m` (향후 30분 평균 출고 지연, 분 단위)
- **시나리오 구조**: 각 시나리오는 정확히 25개 타임슬롯 (15분 간격, 총 약 6시간)

### 결측치

- **86개 컬럼**에 결측 존재 (전체 90개 피처 중)
- 대부분 11.7~13% 수준으로 **균일한 결측률** → 특정 타임슬롯/조건에서 센서 데이터 누락 구조 추정 (MNAR 가능성)
- 결측 비율이 가장 높은 피처: `avg_recovery_time` (13.0%), `congestion_score` (12.9%), `avg_charge_wait` (12.3%)
- LightGBM/XGBoost의 native missing 처리 활용 또는 결측 자체를 indicator 피처로 사용 권장

## 3. 피처 그룹 분류 및 도메인 해석

### 창고 운영 프로세스 흐름

```
주문 접수 → [Order/SKU]
    ↓
로봇 이동·피킹 → [Robot] + [Battery/Charge] + [Congestion/Path]
    ↓
패킹 스테이션 운반 → [Congestion/Path]
    ↓
포장·검수 → [Pack/Manual] + [Worker/Safety]
    ↓
출고 도크 적재 → [Logistics/Warehouse]
    ↓
트럭 출발 → [KPI/Performance]으로 결과 측정
```

## 6. 시계열 패턴 (시나리오 내)

- 각 시나리오: 정확히 **25개 타임슬롯** (15분 간격, 총 약 6시간)
- **타임슬롯 진행에 따른 지연 증가**: TS0 평균 11.3분 → TS24 21.9분 (+93.7%)
- **중앙값은 8.0→9.8분 (+22%)**: 모든 시나리오가 균일하게 악화되는 것이 아니라, **일부 시나리오만 급격히 붕괴** (Std: 11분→36분, 3배 증가)
- 핵심 구동 메커니즘: 배터리 소진의 누적 (→ Section 12 참조)
- **ts_idx, ts_ratio, ts_sin, ts_cos** 피처 추가로 성능 개선 확인 (CV -0.40%)

### 3단계 배터리 캐스케이드 메커니즘

```
[Phase 1: TS 0~6 — 배터리 소모 시작]
로봇들이 완충 상태로 시작 (battery_mean = 68.5%)
→ 활발히 작업 수행 → 배터리 점진적 소모 (→53.4%)
→ 지연 소폭 증가 (11.3 → 17.4분)

[Phase 2: TS 6~12 — 충전 대기열 형성]
일부 로봇 배터리 부족 상태 진입
→ 충전 필요 로봇 급증 (robot_charging: 0.5 → 7.5대)
→ 충전 대기열 형성 (charge_queue: 0 → 3.8대)
→ 가용 로봇 감소 + 남은 로봇에 부하 집중
→ 지연 가속 (→ 19.6분)

[Phase 3: TS 12~24 — 만성적 충전 순환]
배터리 소진이 steady state 진입
→ 충전 완료 로봇 복귀하지만 다른 로봇이 충전 진입
→ 항시 8~9대가 충전 중 (전체의 ~15%)
→ 지연이 높은 수준에서 안정화 (20~22분)
```

### 역설: 혼잡도는 시간에 따라 감소

혼잡도(congestion_score -3.6%), 경로 차단(-20.4%), 고장(-46.0%)이 오히려 감소합니다. **가동 로봇 수 자체가 줄어** 통로가 한산해졌기 때문입니다. 혼잡 감소는 "좋아진 것"이 아니라 "로봇이 줄어서 한산해진 것"입니다.
