## Q1. 어떤 시뮬레이션 기반의 데이터인가?

### 1.1 시뮬레이션이 모사하는 실제 시스템

이 데이터는 **AMR/AGV(자율주행 물류 로봇) 기반 스마트 물류창고**의 약 6시간(25개 타임슬롯, 15분 간격) 운영을 시뮬레이션한 스냅샷입니다. 10,000개의 독립된 시나리오가 250개의 서로 다른 창고 레이아웃(grid, hybrid, narrow, hub_spoke) 위에서 실행되었습니다.

실제로 이런 창고는 쿠팡, 아마존 등의 풀필먼트 센터를 떠올리면 됩니다. 주문이 들어오면 로봇이 선반에서 물건을 집어(picking) 패킹 스테이션으로 운반하고, 사람이 포장해서 출고 도크로 보내는 구조입니다.

### 1.2 컬럼 그룹이 반영하는 창고의 각 부분

데이터의 90개 피처는 11개 그룹으로 나뉘며, 각각 창고 운영의 서로 다른 레이어를 반영합니다:

**① Order/SKU (주문·상품) — 11개 피처**
- 반영 영역: **주문 접수 및 피킹 작업 지시** 단계
- `order_inflow_15m` = 15분간 들어온 주문 수, `unique_sku_15m` = 처리해야 할 고유 물품 수
- `urgent_order_ratio` = 긴급 주문 비율, `heavy_item_ratio` = 무거운 물품 비율
- `sku_concentration` = 특정 물품에 주문 집중도 (높으면 같은 선반에 로봇이 몰림)
- 실제 의미: **"창고가 받아야 할 일의 양과 난이도"**

**② Robot (로봇 운용) — 9개 피처**
- 반영 영역: **AGV/AMR 로봇 함대(fleet)의 현재 상태**
- `robot_active/idle/charging` = 활동/대기/충전 중 로봇 수
- `robot_utilization` = 가동률, `agv_task_success_rate` = 작업 성공률
- `task_reassign_15m` = 작업 재할당 건수 (로봇이 작업 실패 후 다른 로봇에 넘긴 횟수)
- 실제 의미: **"로봇 노동력의 가용 상태"**

**③ Battery/Charge (배터리·충전) — 8개 피처**
- 반영 영역: **로봇 배터리 관리 및 충전 인프라**
- `battery_mean/std` = 전체 로봇의 평균/편차 배터리 잔량
- `low_battery_ratio` = 배터리 부족 로봇 비율 → 사실상 "전투력 상실" 비율
- `charge_queue_length` = 충전 대기열, `avg_charge_wait` = 평균 충전 대기 시간
- 실제 의미: **"로봇이 에너지 고갈 없이 얼마나 일할 수 있는가"** — 지연 예측의 가장 강력한 인자

**④ Congestion/Path (혼잡·경로) — 8개 피처**
- 반영 영역: **창고 통로와 교차로의 물리적 혼잡도**
- `congestion_score` = 종합 혼잡 점수, `max_zone_density` = 가장 밀집된 구역의 밀도
- `blocked_path_15m` = 경로 차단 횟수, `near_collision_15m` = 니어미스(충돌 위험) 횟수
- 실제 의미: **"통로가 막혀서 로봇이 돌아가거나 멈춰야 하는 정도"**

**⑤ Environment (환경) — 14개 피처**
- 반영 영역: **창고 내부 물리 환경** (온도, 습도, CO2, 조명, 진동 등)
- 타겟 상관관계 평균 |r| = 0.019로 거의 무관 → 시뮬레이션에서 노이즈/제어 변수 역할
- 실제 의미: **"극단적 환경이 아닌 한 지연에 직접 영향 미미"**

**⑥ Pack/Manual (패킹·수작업) — 9개 피처**
- 반영 영역: **패킹 스테이션 및 수작업 병목**
- `pack_utilization` = 패킹 라인 가동률, `manual_override_ratio` = 수동 개입 비율
- `staging_area_util` = 스테이징 구역 사용률
- 실제 의미: **"포장·출하 준비 단계의 처리 능력"**

**⑦ Worker/Safety (작업자·안전) — 6개 피처**
- 반영 영역: **현장 인력 및 안전 관리**
- `staff_on_floor` = 현장 인원, `shift_hour` = 교대 시간, `shift_handover_delay_min` = 교대 인수인계 지연

**⑧ Logistics/Warehouse (물류·창고) — 10개 피처**
- 반영 영역: **입출고 도크, 컨베이어, 재고 회전 등 물리적 물류 인프라**
- `loading_dock_util` = 출고 도크 사용률, `conveyor_speed_mps` = 컨베이어 속도

**⑨ System/IT (시스템) — 5개 피처**
- 반영 영역: **WMS(창고관리시스템), 네트워크, 스캐너 등 IT 인프라**
- 타겟 상관관계 평균 |r| = 0.004 → 거의 무관

**⑩ KPI/Performance (성과) — 4개 피처**
- 반영 영역: **창고 운영 KPI** (정시 출고율, 예측 정확도, 역주문 비율)

**⑪ Fault/Recovery (고장·복구) — 5개 피처**
- 반영 영역: **로봇 고장 및 유지보수 시스템**
- `fault_count_15m` = 15분간 고장 건수, `avg_recovery_time` = 평균 복구 시간

### 1.3 데이터가 모사하는 프로세스 흐름

```
주문접수 → [Order/SKU 그룹]
    ↓
로봇이 선반으로 이동 → [Robot + Battery + Congestion 그룹]
    ↓
물품 피킹 → [Robot + Fault 그룹]
    ↓
패킹 스테이션 운반 → [Congestion + Path 그룹]
    ↓
포장·검수 → [Pack/Manual + Worker 그룹]
    ↓
출고 도크 적재 → [Logistics 그룹]
    ↓
트럭 출발 → [KPI 그룹으로 결과 측정]
```

각 단계에서 병목이 생기면 `avg_delay_minutes_next_30m` (출고 지연)이 증가합니다.

### 3.2 이벤트 발생률의 극적 차이

"비정상 이벤트"가 저지연에서는 거의 발생하지 않지만, 고지연에서는 폭발적으로 증가합니다:

| 이벤트 피처 | 저지연에서 발생률 | 고지연에서 발생률 | 차이 |
|------------|-----------------|-----------------|------|
| charge_queue_length > 0 | 1.3% | 44.1% | **34배** |
| avg_charge_wait > 0 | 1.2% | 42.9% | **36배** |
| near_collision_15m > 0 | 1.0% | 42.7% | **43배** |
| blocked_path_15m > 0 | 0.7% | 41.0% | **59배** |
| fault_count_15m > 0 | 1.0% | 35.3% | **35배** |
| task_reassign_15m > 0 | 0.0% | 9.2% | **∞** |

핵심 발견: **저지연 상태에서는 이런 이벤트가 "거의 일어나지 않는 일"이지만, 고지연 상태에서는 "약 40%의 시점에서 동시다발적으로 발생"합니다.** 이는 지연이 단일 원인이 아니라 **여러 문제의 동시 발생(복합 붕괴)**에 의해 일어남을 시사합니다.

### 3.3 극단 지연(P90, ≥45.2분)에서의 상관 역전 현상

흥미로운 발견: 극단적 지연 구간 내부에서는 상관관계가 **역전**됩니다:

| 피처 | 전체 r | 극단 지연 내부 r | 해석 |
|------|--------|-----------------|------|
| pack_utilization | +0.08 | **+0.377** | 극단 구간에선 패킹이 최종 병목 |
| low_battery_ratio | +0.37 | **-0.298** | 배터리로 진입했지만, 내부에선 다른 원인이 지배 |
| battery_mean | -0.36 | **+0.254** | 역전: 극단 구간에선 배터리 높아도 지연 |
| congestion_score | +0.30 | **-0.224** | 혼잡이 아닌 다른 원인 |

해석: **보통 수준의 지연은 배터리·혼잡이 원인이지만, 극단적 지연(45분+)은 이미 그 단계를 넘어서 패킹/출하 단계의 처리 용량 자체가 포화**된 상태입니다. 배터리가 좋아져도 패킹 라인이 병목이면 지연은 계속됩니다.

## Q4. 시간이 지날수록 지연이 증가하는 사유

### 4.1 핵심 발견: 배터리 소진이 시간 지연의 주범

타임슬롯 0(시작)부터 24(약 6시간 후)까지의 변화를 추적한 결과:

| 지표 | TS 0 (시작) | TS 24 (종료) | 변화 | 시간 상관(ρ) |
|------|------------|-------------|------|------------|
| **avg_delay** | 11.3분 | 21.9분 | **+93.7%** | +0.057 |
| **battery_mean** | 68.5% | 48.0% | **-30.0%** | **-0.507** |
| **robot_charging** | 0.5대 | 8.9대 | **+1,522%** | **+0.290** |
| **low_battery_ratio** | 0.000 | 0.172 | 0→17% | **+0.244** |
| **charge_queue_length** | 0.000 | 3.9대 | 0→3.9 | **+0.235** |
| **avg_charge_wait** | 0.000 | 3.5분 | 0→3.5 | **+0.223** |
| robot_idle | 29.4대 | 24.1대 | -18.0% | -0.031 |
| robot_active | 15.3대 | 12.3대 | -19.8% | -0.105 |

### 4.2 시간 경과에 따른 "배터리 캐스케이드" 메커니즘

데이터가 보여주는 시간 열화의 핵심 원인은 **배터리 소진의 연쇄 효과**입니다:

```
[Phase 1: 시작 ~ TS 6]
로봇들이 완충 상태로 시작 (battery_mean = 68.5%)
→ 활발히 작업 수행
→ 배터리가 점진적으로 소모 (→ 53.4%)
→ 지연 소폭 증가 (11.3 → 17.4분)

[Phase 2: TS 6 ~ TS 12]  
일부 로봇이 배터리 부족 상태 진입
→ 충전이 필요한 로봇 급증 (robot_charging: 0.5 → 7.5대)
→ 충전 대기열 형성 (charge_queue: 0 → 3.8대)
→ 가용 로봇 감소 + 남은 로봇에 부하 집중
→ 지연 가속 (→ 19.6분)

[Phase 3: TS 12 ~ TS 24]
배터리 소진이 정상 상태(steady state) 진입
→ 충전 완료 로봇이 복귀하지만 다른 로봇이 충전 진입
→ "충전-복귀-소진" 순환이 지속
→ 상시적으로 8~9대가 충전 중 (전체의 ~15%)
→ 지연이 높은 수준에서 안정화 (20~22분)
```

### 4.3 흥미로운 반전: 혼잡도는 시간에 따라 오히려 감소

| 지표 | TS 0 | TS 24 | 변화 |
|------|------|-------|------|
| congestion_score | 9.8 | 9.4 | -3.6% |
| blocked_path_15m | 0.50 | 0.39 | -20.4% |
| fault_count_15m | 0.46 | 0.25 | -46.0% |

이유 추론: 시간이 지나면서 **가동 로봇 수 자체가 줄어들기 때문에**(active: 15.3→12.3, idle: 29.4→24.1) 통로에 로봇이 적어지고, 혼잡과 충돌도 감소합니다. 즉, **혼잡도 감소는 "좋아진 것"이 아니라 "로봇이 줄어서 한산해진 것"**입니다.

### 4.4 시간-지연 관계의 비선형성

타임슬롯과 지연의 Spearman 상관(ρ=+0.057)이 의외로 약한 이유는:
- 평균은 11→22분으로 2배 증가하지만
- 중앙값은 8.0→9.8분으로 **22%만 증가**
- 표준편차가 11→36으로 **3배 이상 증가**

이는 **모든 시나리오가 균일하게 악화되는 것이 아니라, 일부 시나리오만 급격히 악화**됨을 의미합니다. 시작 조건(배터리, 로봇 수, 레이아웃)이 좋은 시나리오는 끝까지 안정적이지만, 한계에 가까운 시나리오는 시간이 지나면서 "무너지는" 패턴입니다.

## 모델링 시사점 요약

1. **배터리 관련 복합 피처**: `low_battery_ratio × robot_utilization`, `charge_queue_length / robot_active` 등의 교호작용이 핵심
2. **타임슬롯 인덱스**: 시나리오 내 위치를 반드시 피처로 추가 (시간 효과 포착)
3. **이벤트 이진화**: `fault_count > 0`, `blocked_path > 0` 등을 바이너리 플래그로 만들면 70%+ zero-inflation 문제 완화
4. **극단 지연의 별도 모델링**: P90 이상은 패킹 병목이 지배 → 2-stage 모델(일반 구간 + 극단 구간) 고려
5. **시나리오 내 누적 피처**: lag 피처나 rolling 통계가 시간 열화 패턴을 포착할 수 있음
